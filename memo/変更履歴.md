## 2026-07-23: frontend / backend-core の環境構築

- `AGENTS.md` にディレクトリ構成（`frontend/` と `backend-core/` を分離するモノレポ構成）を明記した上で、別々のAIエージェントを並行して割り当てて構築。
- **frontend**: Vite + React + TypeScript。公式の `create-vite` 最新テンプレートを参照し、React 19.2.8 / Vite 8.1.5 / TypeScript 6.0.2（`typescript@latest`は7系だが、公式テンプレートがまだ6系採用のためそちらに合わせた）で構築。`npm install` / `npm run build` / `npm run dev` で動作確認済み。
- **backend-core**: FastAPI。Python 3.14、fastapi 0.139.2 / uvicorn 0.51.0 / sqlalchemy 2.0.51 / asyncpg 0.31.0 / pydantic-settings 2.14.2 / alembic 1.18.5（すべてPyPIで最新版を確認して採用）。`GET /health` を用意し、venv・Docker両方で動作確認済み。
- `docker-compose.yml` を作成し、`db`（PostgreSQL 18）/ `backend-core` / `frontend` の3サービスを結線。PostgreSQL 18でDockerイメージのデータ格納先仕様が変更されていた（`/var/lib/postgresql/data` ではなく `/var/lib/postgresql` をマウントする必要がある）ため対応。
- パッケージのバージョンはすべて一度公式ドキュメント/PyPI/npm registryで確認してから採用する方針で統一。

## 2026-07-24: エージェント・オーケストレーション方式の検討（LangGraph採用）

- CrewAI・LangGraph・自前実装の3つを比較検討。
  - CrewAIは`Crew.kickoff()`がバッチ実行寄りで、会議室・作業室の進行をリアルタイムにストリーミング表示したい要件と相性が悪いため見送り。
  - 自前実装は学習にはなるが、状態管理・ストリーミング・永続化などの「配管」を一から作る必要があり負担が大きい。
  - LangGraphはState/Node/Edgeを自分で明示的に設計する必要があり学習効果を損なわず、かつストリーミング・永続化・human-in-the-loopが標準機能として揃っている点、LangChain本体なしで外部APIを直接ノード内から呼べる点が決め手となり採用を決定。

## 2026-07-24: LangGraphによる最小構成（歩行スケルトン）の実装

- 最初の外部LLM APIとして、実際にAPIキーを持っているDeepSeek APIを採用（OpenAI互換API。`openai` SDKで`base_url`を差し替えて利用）。`deepseek-chat` / `deepseek-reasoner` が公式に非推奨化されるタイミングだったため、後継の `deepseek-v4-pro` を使用。
- `backend-core/pyproject.toml` に `langgraph>=1.2.9` / `openai>=2.47.0` を追加（PyPIで最新版を確認済み）。
- `app/services/llm.py`: DeepSeek呼び出しの薄いラッパーを追加。
- `app/services/graph.py`: 「社長のお題 → 1エージェントがLLM呼び出し → 結果」という一本道のLangGraphグラフ（State/Node/Edge）を追加。DB永続化・複数エージェント・ストリーミングはまだ実装せず、最小構成にとどめた。
- `app/schemas/task.py` / `app/api/tasks.py`: `POST /api/tasks` エンドポイントを追加し、`main.py` にルーター登録。
- 動作確認: venv・Docker Compose（`docker compose build backend-core` で新依存関係を反映）の両方で `GET /health`（デグレなし）と `POST /api/tasks`（日本語のお題を送りDeepSeekから日本語の応答が返る）を確認済み。
- `.env`（実際のAPIキーを含む・gitignore対象）と `.env.example`（コミットされる安全なテンプレート）の役割分担を確認。`DATABASE_URL`のデフォルト値はローカル専用の汎用認証情報のため、`.env.example`に残しても問題ないことを確認済み。

## 2026-07-24: ターミナルからの動作確認手段の追加と.env読み込みの不具合修正

- `curl`でのJSON手打ちが面倒という指摘を受け、`backend-core/scripts/chat.py`を追加。サーバー起動やcurlなしで、LangGraphのグラフを直接呼び出してターミナルで対話しながら動作確認できる簡易CLI。
- `scripts/chat.py`を`backend-core`以外のディレクトリから実行すると `DEEPSEEK_API_KEY` が読み込まれずエラーになる不具合を発見。原因は `app/core/config.py` の `.env` 読み込みが実行時のカレントディレクトリに依存する相対パス（`env_file=".env"`）になっていたこと。`config.py`からの絶対パス（`backend-core/.env`）を明示的に指定する形に修正し、どのディレクトリから実行しても`.env`を読み込めるようにした。
- 動作確認中に、DeepSeekのモデルが自己紹介で「Anthropicが開発したClaudeです」のように別のAIを名乗る現象を確認。通信・実装側のバグではなく、モデル側の自己認識が学習データの影響で混乱する既知の現象。エージェントに正しい自己認識を持たせるには、systemプロンプトで明示的にアイデンティティを伝える必要があることが分かった。今後エージェントの人格・システムプロンプトを設計する際の留意点として記録。
